## 1. Purpose

Define the format and generation rules for the **Suggestions Document** ‚Äî the Pass 1 output of inbox processing. This document contains Tomo's high-level findings with alternatives and confidence scores. The user reviews it to approve the *direction* before Tomo generates detailed instructions in Pass 2.

## 2. Inputs

| Input | Source |
|-------|--------|
| `InboxItemAnalysis[]` | Output of `inbox-analyst` (in-memory) |
| `batch_summary` | Output of `inbox-analyst` (cross-item clusters, date ranges) |
| vault-config | Naming labels, lifecycle prefix |
| Framework profile | Classification names for display |

## 3. Output Location

- **Folder:** `concepts.inbox` (e.g., `+/`)
- **Filename:** `YYYY-MM-DD_HHMM_suggestions.md` ‚Äî current date/time of the `/inbox` run
- **Initial tag:** `#MiYo-Tomo/proposed`

## 4. Document Structure

### Header (frontmatter)

```yaml
---
type: tomo-suggestions
generated: 2026-04-08T14:30:00Z
tomo_version: "0.1.0"
profile: miyo
source_items: 5
tags:
  - MiYo-Tomo/proposed
---
```

### Title section

```markdown
# Tomo Suggestions ‚Äî 2026-04-08 14:30

## Batch Summary
- Analyzed 5 inbox items spanning 2026-04-07 to 2026-04-08
- 2 coding insights, 2 fleeting notes, 1 task
- 1 topic cluster detected: "shell/terminal" (3 items) ‚Üí potential new MOC

## How to use this document
Review each suggestion below. For each item:
- Leave checked (`[x]`) to approve as-is
- Change fields below the suggestion to modify
- Uncheck (`[ ]`) to skip (item stays in inbox as `captured`)
- Pick an alternative by checking its box instead

When finished, change this document's tag from `#MiYo-Tomo/proposed` to
`#MiYo-Tomo/confirmed`, then run `/inbox` again.
```

## 5. Action ID Numbering

Each suggestion gets a stable ID (`S01`, `S02`, `S03`, ...):
- Used by Pass 2 to reference user's choices
- User can reference them in discussions ("S03 looks wrong")
- IDs are local to this suggestions document ‚Äî not cross-run unique

## 6. User Interaction Model

The user has three main choices per item, plus modification options:

### Tri-State Per Item

Each suggestion has two checkboxes ‚Äî one for Approve, one for Delete:

```markdown
- [x] Approve     ‚Üê checked by default (Tomo's recommendation)
- [ ] Delete source    ‚Üê unchecked by default; only relevant if Approve is unchecked
```

| State | Approve | Delete | What happens |
|-------|---------|--------|--------------|
| **Approve** | `[x]` | ignored | Item proceeds to Pass 2 instruction set |
| **Skip** | `[ ]` | `[ ]` | Item stays in inbox as `captured`, re-processed next run |
| **Delete** | `[ ]` | `[x]` | Source inbox file is deleted during cleanup |

The Delete checkbox is only meaningful when Approve is unchecked. If both are checked, Approve wins.

### Modification Options

| Intent | How to express |
|--------|---------------|
| Approve as-is | Leave `[x] Approve` checked |
| Skip (keep for later) | Uncheck `[ ] Approve`, leave Delete unchecked |
| Delete source file | Uncheck Approve, check `[x] Delete source` |
| Pick alternative | Uncheck primary, check an alternative |
| Modify specific field | Edit the field text below the suggestion (e.g., change title) |
| Combine multiple actions | Approve primary AND also check tracker update sub-action |

**Tomo reads the user's modifications via markdown parsing:**
- Checkbox state on `- [x] Approve` / `- [ ]` lines
- Field values on `- Key: value` lines below the suggestion
- Any freeform notes the user added are ignored unless in a recognized pattern

## 7. Pass 2 Handoff

When the user changes the document tag from `#MiYo-Tomo/proposed` to `#MiYo-Tomo/confirmed`, the next `/inbox` run picks it up and:

1. Reads the document
2. Parses each S## section for checkbox state and field modifications
3. Builds a list of confirmed actions per item
4. Hands off to `instruction-builder` for Pass 2

Items where nothing is checked are silently skipped (left in inbox as `captured` for future runs).

## 8. Styling and Readability

The document is meant to be read in Obsidian:
- Use Obsidian wikilinks `[[...]]` for all references (clickable navigation)
- Use H2/H3 hierarchy so Obsidian's outline pane is useful
- Keep per-item section <50 lines ‚Äî reviewable in one screen
- Group related info tightly, avoid long paragraphs
- Use bold for field labels, regular text for values

## 9. Document Length Limits

If the batch is huge (>50 items), the document becomes unwieldy. Mitigation:
- **Soft limit:** 30 items per document. Beyond that, split into multiple documents (`_suggestions-part1.md`, `_suggestions-part2.md`).
- **Hard limit:** none ‚Äî but warn the user during generation
- User can request smaller batches by running `/inbox --limit 20`

## 10. Error Handling During Generation

- **Item with missing analysis data:** include in document with placeholder "‚öÝÔ∏è Analysis failed, skip" notice; mark the issue in an Issues section at the top
- **Cluster detection confused:** skip the cluster section, include items individually
- **All items skipped:** generate an empty document with only the Batch Summary; tag still set to `proposed` so the user can still confirm emptiness
