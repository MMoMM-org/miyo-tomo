## 1. Purpose

Define the schema for vault-config.yaml — the user's ground truth. Generated by the setup wizard, refined by `/explore-vault`. Overrides all profile defaults.

## 2. Role in Knowledge Stack

Layer 3 is the **authority**. It always wins:
- **L3 field present** → use L3 value
- **L3 field omitted** → fall back to L2 profile default
- **L3 field set to `null`** → intentionally disabled, no fallback

## 3. Schema Overview

Two notable structural decisions reflected below:

1. **Calendar config splits granularity (location) from naming pattern.** Folder paths live in `concepts.calendar.granularities`, filename patterns live in `naming.calendar_patterns`. The pattern is "naming" — it shares the conceptual space with display labels.

2. **map_note discovery supports both paths AND tags.** Top-level (Dewey) MOCs may live in one folder, but sub-MOCs can be scattered throughout the vault. A tag like `type/others/moc` finds them regardless of folder. Tomo uses both signals to build the MOC hierarchy.

**MOC tree:** LYT and MiYo start from a HomeNote into the Dewey classification level, then branch into multiple layers of sub-MOCs. Tomo must detect and organize MOCs at all levels — not just classification-level. See [LYT/MOC Linking workflow](../workflows/lyt-moc-linking.md).

```yaml
schema_version: integer        # Validated at session start

# Identity
profile: string                # Which profile to load (miyo, lyt, para, custom)
profile_version: string

# Concept mapping (overrides profile defaults)
concepts:
  inbox: string

  atomic_note: string

  map_note:
    paths: string[]            # One or more folders where MOCs live
    tags: string[]             # Tags that identify a note as a MOC,
                               # e.g. ["type/others/moc"]
                               # Used as a fallback / supplement to paths

  calendar:
    base_path: string
    granularities:
      daily:    { enabled: bool, path: string }
      weekly:   { enabled: bool, path: string }
      monthly:  { enabled: bool, path: string }
      quarterly: { enabled: bool, path: string }
      yearly:   { enabled: bool, path: string }

  project: string
  area: string
  source: string
  template: string
  asset: string

# Naming (labels for display + filename patterns)
naming:
  labels:                      # How Tomo refers to things in proposals
    map_note: string
    atomic_note: string
    inbox_item: string
    project: string
    source: string
  calendar_patterns:           # Filename pattern per granularity
    daily: string              # e.g. "YYYY-MM-DD"
    weekly: string             # e.g. "YYYY-[W]ww"
    monthly: string            # e.g. "YYYY-MM"
    quarterly: string          # e.g. "YYYY-[Q]Q"
    yearly: string             # e.g. "YYYY"

# Lifecycle tracking
lifecycle:
  tag_prefix: string           # Default: "MiYo-Tomo"

# Template mapping
templates:
  base_path: string
  mapping:
    atomic_note: string
    map_note: string
    daily: string
    weekly: string
    monthly: string
    yearly: string
    project: string
    source: string

# Frontmatter schema → Tier 3: frontmatter-schema.md
frontmatter:
  required: FieldDef[]
  optional: FieldDef[]

# Relationships → Tier 3: relationship-config.md
relationships:
  parent: { marker: string, format: string, position: string }
  peer: { marker: string, format: string, position: string }

# Tag taxonomy → Tier 3: tag-taxonomy.md
tags:
  prefixes: { [prefix]: string[] }

# Callout mapping → Tier 3: callout-mapping.md
callouts:
  editable: { [name]: string }
  protected: { [name]: string }

# Protected patterns
protected_patterns: string[]
```

## 4. Schema Versioning

- `schema_version: 1` for MVP
- Validated at Tomo session start
- If Tomo requires a newer schema than the config provides:
  1. Present diff of what changed
  2. Guide user through migration
  3. Block operation until migration complete
- Migration is non-destructive: old fields preserved, new fields added with defaults

## 5. Generation Flow

### Phase 1: Install Script
Produces minimal config with: profile, concept folder mappings, lifecycle prefix, schema_version.

### Phase 2: First-Session Discovery
Enriches config with: frontmatter schema, tag taxonomy, callout mapping, relationship conventions, template mapping.

### Manual Editing
Users can edit vault-config.yaml directly. Tomo validates on next session start.

## 6. Validation Rules

At session start, Tomo validates:
1. `schema_version` is supported
2. `profile` exists in available profiles
3. All required concept paths are non-null
4. Template files referenced in `mapping` exist (via Kado)
5. No conflicting field definitions in frontmatter schema

Validation failures produce clear error messages with fix instructions. Tomo does not proceed with invalid config.
