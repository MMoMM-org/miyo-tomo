# Tier 3: Staleness Policy

> Parent: [Discovery Cache](../../tier-2/components/discovery-cache.md)
> Status: Implemented

---

## 1. Purpose

Define how Tomo detects, reports, and handles stale discovery cache data. The cache is advisory — Tomo must function without it, but functions better with fresh data.

## 2. Staleness Detection

Cache age is computed from the `last_scan` timestamp in `discovery-cache.yaml`:

```
cache_age_days = (current_time - last_scan) / 86400
```

### Thresholds

| Age | Status | Tomo behavior |
|-----|--------|---------------|
| 0-3 days | Fresh | Full confidence in cache data |
| 4-7 days | Aging | Normal operation, log info-level note |
| 8-14 days | Stale | Warn user at start of `/inbox`: "Cache is N days old. Run `/explore-vault` for better results." |
| 15+ days | Very stale | Warn + degrade: skip confidence scores from cache, rely on profile keywords only |
| No cache file | Missing | Full degraded operation (see §4) |

### Where Checked

- **Session start:** Tomo validates cache exists and age; reports if stale
- **`/inbox` start:** Before analysis begins; warns if stale
- **Individual MOC lookups:** If a specific MOC from cache can't be found via Kado → treat as stale entry

## 3. Staleness Triggers Beyond Age

Age isn't the only staleness signal:

| Trigger | How detected | Impact |
|---------|-------------|--------|
| Cache age | Timestamp comparison | See thresholds above |
| MOC not found | `kado-read` on cached MOC path returns 404 | Remove from cache in-memory; warn |
| Note count divergence | `kado-search listDir` returns significantly different count than `scan_stats.total_notes` | Suggest refresh |
| New tags observed | Inbox analysis finds tags not in `tag_patterns` | Informational only |

**Note count divergence threshold:** if observed count differs by >20% from cached count, the cache is likely stale.

## 4. Degraded Operation (No Cache)

When the cache is missing or too stale to trust:

| Component | Normal (with cache) | Degraded (without cache) |
|-----------|--------------------|-----------------------|
| MOC matching | Compare against cached topics | Read MOCs on-demand via Kado per inbox item |
| Classification | Use enriched keywords from cache | Profile keywords only (seed set) |
| Tag suggestions | Use observed tag patterns from cache | Use tag taxonomy from config only |
| Orphan detection | Check cache orphan list | Skipped entirely |
| New MOC detection | Cross-reference cached notes | Skipped (no baseline to compare) |
| Performance | Fast (in-memory lookups) | Slow (N kado-reads per inbox item per MOC) |

**Critical point:** Tomo is fully functional without a cache. It's just slower and less accurate. The cache is an optimization, not a requirement.

## 5. On-Demand MOC Reading (Fallback)

When cache is unavailable for MOC matching, inbox-analyst falls back to:

```
1. Get list of MOC paths from vault-config (concepts.map_note.paths + tags)
2. For each MOC path:
   kado-read the note content
   Extract title + H2 sections + topics (inline, not cached)
3. Match inbox item topics against live-read MOC topics
4. Return matches with confidence
```

**Performance impact:** If there are 30 MOCs and 10 inbox items, this is 300 kado-reads instead of 0 (with cache). At ~50ms per read, that's ~15 seconds vs near-instant.

## 6. Cache Refresh

Cache is refreshed **only by explicit user action**: `/explore-vault`

No automatic background refresh. Reasons:
- Tomo runs in Docker — no persistent daemon
- Each `/inbox` invocation is a discrete session
- Kado may not be available between sessions
- User controls when vault knowledge is updated

**Recommendation to user:** Run `/explore-vault` weekly, or after significant vault changes (major reorganization, many new notes, new MOCs created).

## 7. Cache Version Mismatch

If `cache_version` in the file doesn't match Tomo's expected version:
- Treat entire cache as stale
- Warn: "Cache was generated by a different Tomo version. Run `/explore-vault` to rebuild."
- Fall back to degraded operation

This prevents bugs from reading cache data with an unexpected schema.

## 8. Interaction with Auto-Detection

The automated applied detection feature (in `instruction-set-cleanup.md §5`) reads target files directly via Kado — it does NOT use the discovery cache. This means auto-detection works regardless of cache freshness.
