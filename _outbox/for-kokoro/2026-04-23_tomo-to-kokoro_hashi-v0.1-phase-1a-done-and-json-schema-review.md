## Phase 1b — `.json` sibling (already shipped; schema review requested)

### Status

Already in production. `instruction-render.py` v0.5.0 writes both
`instructions.json` (canonical, machine-readable) and `instructions.md`
(human-review artifact rendered deterministically from the JSON) into
`--output-dir`. The `instruction-builder` agent then Kado-writes both
to the inbox folder with the same timestamp prefix:

- `<inbox><YYYY-MM-DD_HHMM>_instructions.md` — operation=note
- `<inbox><YYYY-MM-DD_HHMM>_instructions.json` — operation=file
  (base64, via `scripts/kado-write-file.py`)

Behavior matches every must-have in your handoff:

- Same folder, same timestamp. ✓
- `.md` stays the human-review artifact, tagged
  `#MiYo-Tomo/instructions`. ✓
- `.json` decoupled from `.md` wording. ✓
- No LLM prompts in the JSON — `instruction-render.py` composes it
  from `build_actions()` deterministically. ✓
- No conditional logic — all branches are resolved at generation time
  (e.g. `position` is `after_last_line` · `before_first_line` ·
  `at_time`, not "figure it out"). ✓
- No Kado API references in the schema. ✓
- Metadata links to the `.md` peer via shared timestamp +
  `source_suggestions`. ✓
- Schema version present. ✓

### Documentation

Reference spec updated in the same branch as Phase 1a:

- `docs/XDD/reference/tier-3/inbox/instruction-set-generation.md` §11

### Schema — top-level document

```json
{
  "schema_version": "1",
  "type": "tomo-instructions",
  "source_suggestions": "YYYY-MM-DD_HHMM_suggestions",
  "generated": "2026-04-23T14:35:00Z",
  "profile": "miyo",
  "tomo_version": "0.x.y",
  "action_count": 31,
  "actions": [ /* ordered */ ]
}
```

### Schema — action types

All actions carry `id` (`I01`, `I02`, …, matching `.md` headings) and
`action` (discriminator). Execution order inside the array:

1. `create_moc` — new MOC first so later links can target it
2. `move_note` — atomic notes from inbox to destination
3. `link_to_moc` — parent-MOC up-links and supporting-items down-links
4. `update_tracker` / `update_log_entry` / `update_log_link` — daily updates
5. `delete_source` — inbox cleanup once content is captured elsewhere
6. `skip` — explicit no-op with a reason

Field-by-field catalog is in the spec §11.2. The condensed version for
your review is below — what Hashi receives for each action type:

- **create_moc** — `title`, `rendered_file`, `source`, `destination`,
  `parent_moc`, `supporting_items`, `tags`, `template`.
- **move_note** — `title`, `rendered_file`, `source`, `destination`,
  `origin_inbox_item`, `parent_mocs`, `tags`.
- **link_to_moc** — `target_moc`, `target_moc_path` (Kado-resolved by
  Tomo at generation time, `null` only for in-set create_moc targets),
  `section_name` (first line of target callout, `null` for in-set
  MOCs — Hashi must read the rendered create_moc's own callout at
  execute time), `source_note_title`, `line_to_add` (literal).
- **update_tracker** — `date`, `daily_note_path`, `field`, `value`,
  `syntax` (`inline_field` · `task_checkbox` · `frontmatter`),
  `section`, `source_stem`, `reason`.
- **update_log_entry** — `date`, `daily_note_path`, `section`,
  `heading_level`, `position` (`after_last_line` ·
  `before_first_line` · `at_time`), `time` (required if
  `position=at_time`), `content`, `source_stem`, `reason`.
- **update_log_link** — same as update_log_entry but `content` →
  `target_stem`.
- **delete_source** — `source_path`, `reason`.
- **skip** — `source_path` (optional), `reason`.

### Known schema decisions that might warrant your review

Surfacing these explicitly so the review doesn't miss them:

1. **`link_to_moc.section_name` is nullable when the target MOC is in
   the same instruction set.** Tomo can't pre-read a file it's about
   to create. The rendered `create_moc` template provides its own
   editable callout; Hashi must read it at execute time. Alternative:
   Tomo could hardcode the callout name from the template into the
   action at generation time, but then a template change would silently
   drift. Current design prefers "let Hashi look it up" over
   "bake-in and risk drift". Your call.

2. **`line_to_add` is the literal text including the leading `- `.**
   So `- [[Note Title]]` goes in verbatim. No marker, no separate
   target stem. Rationale: Hashi doesn't need to know it's a list
   item; it's pure text insertion at the resolved position. If you
   want a structured form (`list_marker: "- "` + `target_stem:
   "Note Title"`), this is the cheap time to ask.

3. **`create_moc.supporting_items` is a count, not a list.** The
   actual child items each get their own `link_to_moc` action in the
   same set. The field is informational for humans ("this MOC will
   get 5 link actions below"). If Hashi wants it as the real list
   instead of the count, that's a schema change — flag it.

4. **Tag `MiYo-Tomo/state/*` values flow through `tags`** on
   `create_moc`/`move_note`. Hashi gets them as-is; state-tag
   rewriting to `applied` is lifecycle-executor territory
   (vault-executor, unchanged by this work).

5. **No explicit `.md_peer` field.** The peer is implied by same
   folder + same timestamp + matching stem. If Hashi wants an explicit
   field to avoid ambiguity, trivial to add.

6. **Version bumps.** `schema_version: "1"` stays as long as changes
   are additive. Breaking changes bump it and come back through this
   handoff channel before landing.
